# Privacy and Compliance Boundaries

This project is intentionally designed with strict privacy boundaries.

## What This Project Uses

- Synthetic patient names
- Synthetic patient IDs
- Synthetic accession numbers
- Synthetic exam descriptions
- Synthetic timestamps
- Synthetic system names
- Synthetic message payloads

## What This Project Does Not Use

- No real PHI
- No workplace screenshots
- No real logs
- No real credentials
- No real server names
- No internal documentation
- No production system data
- No real patient identifiers
- No real diagnostic imaging messages

## Compliance Positioning

This is a healthcare-inspired portfolio and learning project. It is not a production clinical system.

The project may discuss privacy-aware design concepts such as:

- access control
- audit history
- raw message retention
- failure isolation
- least privilege
- synthetic data usage
- protected dashboard access

However, it does not claim PHIA, HIPAA, SOC 2, or production compliance.

## Recommended Public Wording

Use:

> This project is healthcare-inspired and uses synthetic data only. It demonstrates cloud-native architecture patterns for workflow reconciliation, observability, and failure handling, but it is not a production-compliant healthcare system.

Avoid:

> This project is HIPAA compliant.

Avoid:

> This project uses real hospital workflows/data.
